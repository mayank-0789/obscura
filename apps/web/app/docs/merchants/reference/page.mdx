# API reference

Every export from `@obscura-app/merchant-sdk`, one section per symbol.

## `payrail(config)`

The merchant-side factory. Returns a `PayrailMerchantClient` — a small
object whose only method is `charge()`.

<CodeBlock lang="ts">
{`import { payrail } from "@obscura-app/merchant-sdk";

const pay = payrail(config: MerchantSdkConfig);`}
</CodeBlock>

## `pay.charge(config)`

Returns an Express-compatible middleware. Runs *before* your route
handler, intercepts payment-less requests with 402, and settles
payment-bearing requests before calling `next()`.

### Side effects on `res`

On a successful settlement, the middleware:

- Sets an `X-Payment-Response` header — base64-encoded JSON of the
  settlement result.
- Sets `res.locals.payrailSettlement` — the parsed settlement object
  with the Solana tx signature.
- Calls `next()` so your downstream handler runs with both in place.

See [Settlement receipts](/docs/merchants/receipts) for reading them.

## Re-exports

- **`PaymentRequirements`** — the x402-solana type for a 402 challenge
  body. Useful for typing handlers that read settlement via
  `res.locals.payrailSettlement`.

<CodeBlock lang="ts">
{`import type { PaymentRequirements } from "@obscura-app/merchant-sdk";`}
</CodeBlock>

## Protocol-level notes

- **One `createPaymentRequirements` call per request.** The SDK dedupes
  the `/supported` round-trip internally, so verify + settle reuse the
  same quote built during the 402 phase.
- **Synchronous settle.** The middleware waits for on-chain confirmation
  before calling `next()`. See [How it works](/docs/merchants/flow) for
  the latency trade-off.
- **No Payrail API calls from the middleware.** The SDK talks directly
  to the PayAI facilitator + Solana RPC — zero dependency on Payrail
  infrastructure at runtime.
